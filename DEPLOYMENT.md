# Deployment Checklist

This checklist prepares the repository for GitHub, Render, Supabase/Postgres, and Telegram webhook operation.

## 1. GitHub

Repository target:

```text
https://github.com/nrdex42-ctrl/Anti_Detection_FB_Automation.git
```

Before pushing, verify no secrets are tracked:

```bash
rg -n --hidden "datr=|c_user=|xs=|fr=|TELEGRAM_BOT_TOKEN=.+|DATABASE_URL=postgres|ENCRYPTION_KEY=.+" \
  --glob '!artifacts/**' \
  --glob '!diagnostics/**' \
  --glob '!logs/**' \
  --glob '!__pycache__/**' \
  --glob '!.git/**' .
```

## 2. Supabase/Postgres

Create a Supabase project and copy a Postgres connection string into Render as `DATABASE_URL`.

For Render or any IPv4-only hosting network, prefer the Supabase Session Pooler connection string. Supabase documents direct and pooler Postgres connection options here:

```text
https://supabase.com/docs/guides/database/connecting-to-postgres
```

Schema options:

- Leave `AUTO_INIT_DB=true` so the bot applies `supabase/schema.sql` on startup.
- Or run the schema manually:

```bash
export DATABASE_URL="postgresql://..."
python scripts/init_supabase.py
```

The schema creates:

- `fb_accounts` for encrypted account cookies.
- `fb_pages` for discovered managed pages.
- `fb_post_jobs` for Telegram queued posts.
- `fb_account_runtime` for account lock leases and cookie cooldown state.
- `telegram_user_state` for each Telegram user's active Facebook account selection.

## 3. Telegram Bot

Create a bot with BotFather and copy the token into Render as `TELEGRAM_BOT_TOKEN`.

Set a strong `TELEGRAM_WEBHOOK_SECRET`. Telegram sends this value in the `X-Telegram-Bot-Api-Secret-Token` header when configured through `setWebhook`.

Official Bot API reference:

```text
https://core.telegram.org/bots/api#setwebhook
```

After Render gives you a public URL, automatic webhook setup uses Render's `RENDER_EXTERNAL_URL` by default. Set `PUBLIC_BASE_URL` only if you use a custom domain or need to override the Render URL:

```bash
export TELEGRAM_BOT_TOKEN="..."
export TELEGRAM_WEBHOOK_SECRET="..."
export PUBLIC_BASE_URL="https://your-custom-domain.example"
python scripts/set_telegram_webhook.py
```

## 4. Render

The repository includes `render.yaml`. Render Blueprints support `runtime: python`, `buildCommand`, and `startCommand`.

Official Blueprint reference:

```text
https://render.com/docs/blueprint-spec
```

`render.yaml` marks these required Render environment variables as `sync: false`. Set them once in the Render dashboard; future Blueprint deploys will not overwrite them:

```text
TELEGRAM_BOT_TOKEN=<BotFather token>
BOT_ADMIN_IDS=<your Telegram numeric user id>
DATABASE_URL=<Supabase Session Pooler URL>
ENCRYPTION_KEY=<stable Fernet key; keep the same value for the same Supabase database>
```

`PUBLIC_BASE_URL` is optional for a normal Render `onrender.com` deployment because the app reads Render's runtime `RENDER_EXTERNAL_URL`. If you move the project to another Render account and the hostname changes, the app will ignore a stale `PUBLIC_BASE_URL` when both values are `onrender.com` URLs. Set `PUBLIC_BASE_URL_FORCE=true` only if you intentionally need to force the manual value.

`TELEGRAM_WEBHOOK_SECRET` is generated by Render from the Blueprint.
If Render generates symbols that Telegram does not allow in `TELEGRAM_WEBHOOK_SECRET`, the app derives a stable Telegram-safe header token automatically.

`ENCRYPTION_KEY` must be stable. The bot encrypts stored Facebook cookies with this value before saving them to Supabase. If you move to another Render account or recreate the service while reusing the same Supabase database, copy the old `ENCRYPTION_KEY` into the new service. If the old key is lost, existing encrypted cookies cannot be recovered; re-add the Facebook accounts/cookies in the bot so they are encrypted with the new key.

## 5. Production Operation

1. Deploy the Render service from the GitHub repository.
2. Confirm Render logs include `Telegram webhook configured for https://.../telegram/webhook` and that the hostname matches Render's current primary URL.
3. Open `/healthz` and confirm `telegram_webhook_base_url` matches the current Render service URL.
4. Send `/start` to the bot.
5. Use the typing-area dashboard panel to add an account.
6. Use `Refresh Pages` / `Discover Pages` to store managed pages for that account.
7. Use `Switch Active Account`, quick post buttons, or `Post to All Pages` to queue posts through guided steps.

If automatic webhook setup is disabled, set it manually:

```bash
export TELEGRAM_BOT_TOKEN="..."
export TELEGRAM_WEBHOOK_SECRET="..."
# Optional for custom domains. On Render, RENDER_EXTERNAL_URL is used automatically.
export PUBLIC_BASE_URL="https://your-custom-domain.example"
python scripts/set_telegram_webhook.py
```

The dashboard Add Facebook Account button accepts raw cookie headers, JSON cookie arrays, `{cookies:[...]}` exports, uploaded JSON files, and multi-message JSON chunks followed by the `✅ Done` button.

Page discovery is cached in `fb_pages`. Normal posting reads the saved page list and does not rediscover pages before each post. Use the `🔄 Refresh Pages` dashboard button when you want to refresh a selected account's cached pages.

Normal users can use `/start` and the typing-area dashboard. Their accounts, cached pages, and post jobs are scoped to their Telegram user id.

Admins listed in `BOT_ADMIN_IDS` can use `🔒 Admin Dashboard` to view system stats, users, accounts, post stats, active account locks, and runtime config.

Set `RESTART_BROADCAST_ENABLED=true` to let the bot send known users a refreshed dashboard message once per Render deploy revision.

## 6. Isolation Rules

- Different accounts can process concurrently.
- The same account is protected by a Supabase lock lease.
- After a posting attempt uses a cookie, the bot records `last_cookie_used_at`.
- `BOT_ACCOUNT_COOKIE_COOLDOWN_SECONDS` controls the minimum wait before the next use of the same account.
- `BOT_ACCOUNT_LOCK_HEARTBEAT_SECONDS` refreshes long-running job leases.
